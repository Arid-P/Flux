This document defines the full technical architecture, technology stack, database schema (v2), module boundaries, routing, API design, performance requirements, and development checklist for FluxDone v2. It incorporates all changes from the full UI specification phase. All deviations from this document require explicit approval and must be documented.
2. Changes from TRD v1.0
2.1 Tech Stack Changes
Hive removed from tech stack entirely. SQLite (sqflite) is the sole local storage for all relational data. shared_preferences handles lightweight key-value settings.
flutter_bloc replaces Riverpod as the state management library. All feature modules use BLoC/Cubit pattern.
injectable added alongside get_it for compile-time dependency injection code generation.
flutter_slidable added for swipe actions on task cards and trash cards.
confetti added for habit completion celebration animation.
2.2 Database Schema Changes
habits table: two new columns added — target_count (INTEGER NOT NULL DEFAULT 1) and icon_identifier (TEXT NOT NULL DEFAULT 'star').
habits table: new column reminder_time (INTEGER NULLABLE) for storing reminder time-of-day as Unix ms. Notification scheduling is Phase 2 but column exists from Phase 1.
tasks table: new column is_pinned (INTEGER NOT NULL DEFAULT 0). Pin UI is Phase 2 but column exists from Phase 1 to avoid a breaking schema migration later.
folders table: no color_hex column in v1. Explicitly documented. May be added in Phase 2 via migration.
All timestamps remain as INTEGER (Unix epoch milliseconds, UTC).
2.3 Architecture Changes
Folder management (SCR-10) and List management (SCR-11) are not navigation routes. No go_router routes for /folder/manage or /list/manage. All management is inline in the side drawer via AlertDialog and showModalBottomSheet.
JSON theming architecture: all color values in the widget tree must be sourced from a ThemeTokens class (not hardcoded). This enables Phase 3 JSON theming with no widget rewrites.
FluxFoxus bridge MethodChannel name: com.fluxfoxus/fd_integration (corrected from v1.0 which referenced FluxFocus).
2.4 Route Map Changes
Removed routes: /folder/manage, /list/manage (inline in drawer, not routable).
Added route: /task/recurrence/custom (custom recurrence screen from SCR-02 recurrence picker).
Added route: /settings/google/backup (SCR-13 Google Drive Backup, accessed from Settings → Google sub-screen).
3. System Architecture
3.1 Architecture Pattern
FluxDone uses Clean Architecture with a feature-first folder structure and BLoC/Cubit state management. Each feature module is self-contained: its own presentation layer (Widgets, Screens, BLoC/Cubit), domain layer (entities, use cases, repository interfaces), and data layer (repository implementations, DAOs). Cross-module communication uses a shared repository interface layer and a pub/sub event bus via rxdart. No feature module may import another module's internal files.

5. Database Schema v2
5.1 Schema Design Principles
All primary keys: INTEGER PRIMARY KEY AUTOINCREMENT (except presets which use TEXT UUID).
All timestamps: INTEGER (Unix epoch milliseconds, UTC). Application layer converts to local time.
Booleans: INTEGER (0 = false, 1 = true).
Nullable fields explicitly marked. Non-nullable fields have NOT NULL constraints.
Soft deletes on tasks: is_trashed flag + trashed_at timestamp. Hard delete only from Trash purge.
Recurrence rules: JSON string in a single column (avoids separate recurrence table in Phase 1).
All foreign keys use ON DELETE CASCADE.
No Hive boxes — shared_preferences handles all key-value settings.

5.10 Recurrence Rule JSON Schema
The recurrence_rule column in the tasks table stores a JSON string. All valid structures:

{ "type": "daily" }
{ "type": "weekly", "days": [1, 3, 5] }              // 0=Sun, 1=Mon, ... 6=Sat
{ "type": "monthly_date", "day": 15 }                // Every 15th of month
{ "type": "monthly_weekday", "week": 2, "day": 1 }   // Every 2nd Monday
{ "type": "yearly" }                                  // Same date each year
{ "type": "interval", "every_n_days": 3 }            // Every 3 days

Recurrence edit scope (applied at application layer, not stored in DB):
this_only — Detach instance from chain (set recurrence_parent_id = null, clear recurrence_rule), apply changes.
this_and_future — Update this instance and regenerate all future instances from this date.
all — Update parent task and regenerate all instances past and future.

7. Token-Based Theming Architecture
7.1 Requirement
All color values used anywhere in the widget tree must be sourced from ThemeTokens. No color value may be hardcoded inside a Widget's build method or constructor. This is mandatory from Phase 1 to enable Phase 3 JSON custom theming with no widget rewrites.

7.2 ThemeTokens Class
class ThemeTokens {
  // Background & surface
  final Color background;
  final Color surface;
  final Color surfaceVariant;
  // Text
  final Color textPrimary;
  final Color textSecondary;
  // Accent / primary
  final Color primary;
  final Color primaryContainer;
  // Semantic
  final Color danger;       // #E53935
  final Color warning;      // #FB8C00
  final Color success;      // #43A047
  // Dividers & borders
  final Color divider;
  final Color border;
  // Domain colors (system-locked)
  final Color numberTheory;    // #2E7D32
  final Color geometry;        // #1565C0
  final Color combinatorics;   // #43A047
  final Color algebraP1;       // #FB8C00
  final Color algebraP2;       // #E64A19
  final Color tests;           // #E53935
  final Color neevDiamond;     // #576481
}

ThemeTokens is provided to the widget tree via InheritedWidget or Theme.of(context) extension. All widgets access colors via context.tokens.primary, context.tokens.surface, etc. Domain colors are always sourced from ThemeTokens even when user cannot change them (this enables future JSON override).

7.3 Phase 3 JSON Theming
When Phase 3 is implemented, a JSON file maps token names to hex values. The app parses the JSON and constructs a ThemeTokens instance from it. No widget code changes required. The JSON schema mirrors the ThemeTokens field names exactly.
8. FluxFoxus Bridge (MethodChannel IPC)
8.1 Channel Name
const fdChannel = MethodChannel('com.fluxfoxus/fd_integration');

8.2 FD → FF: FocusBlockRequest
When a FD task with both start_time and end_time is created, updated, or deleted, FD sends a FocusBlockRequest to FF.

// FD side: send FocusBlockRequest
await fdChannel.invokeMethod('focusBlockRequest', {
  'taskId': task.id.toString(),
  'taskName': task.title,
  'startTime': task.startTime?.millisecondsSinceEpoch,
  'endTime': task.endTime?.millisecondsSinceEpoch,
  'duration': task.duration?.inSeconds,
  'listId': task.listId.toString(),
  'sectionId': task.sectionId?.toString(),
  'action': action.name, // 'create', 'update', or 'delete'
});

Recurring task instances: each new instance generated by FD fires a new FocusBlockRequest (CREATE action) to FF independently.

8.3 FF → FD: createTask
FF calls createTask on this channel when a user starts a focus session from FF (not triggered from FD). FD writes a 'Focus Session' task to the 'FF' section of the default list.

// FD side: handle createTask call from FF
fdChannel.setMethodCallHandler((call) async {
  if (call.method == 'createTask') {
    final args = call.arguments as Map;
    await taskRepository.createTask(Task(
      title: 'Focus Session',
      description_md: args['description'],
      listId: defaultListId,
      sectionId: ffSectionId, // 'FF' section in default list
      taskDate: DateTime.now(),
    ));
  }
});

The 'FF' section must exist in the default list from initial app setup. FD creates this section automatically on first launch if it does not exist. The user can rename it but the section identifier (sectionId) is stored in shared_preferences so FF can always resolve it.

12. Technical Success Criteria
Phase 1 is technically complete when all of the following are true:

All P0 and P1 features pass manual end-to-end testing on a physical Android device (API 26+) with zero network connectivity.
SQLite database survives app kill, device restart, and OS-level process termination without data loss or corruption.
No feature module imports another module's internal files. Module isolation enforced and verified by code review.
No hardcoded color values exist anywhere in the widget tree. All colors sourced from ThemeTokens.
All database writes use parameterized sqflite queries. No raw SQL string concatenation anywhere.
Recurring task generation produces correct instances for all 6 recurrence types across month boundaries and leap years.
Calendar timeline correctly handles: single task, 2 overlapping tasks, 3 overlapping tasks, task spanning midnight, date-only tasks (no time).
Auto-save (500ms debounce) correctly batches rapid changes and does not lose data on sheet dismiss.
Light and dark themes apply correctly on every screen and sub-screen. No color token is missing from either ThemeTokens set.
Local notifications fire within 1 second of scheduled time on a physical device with app in background and screen off.
App performance meets all targets defined in Section 10 on a mid-range Android device.
FD sends FocusBlockRequest to FF correctly on task create/update/delete (with start+end time) when both apps are installed.
FD receives createTask from FF and writes 'Focus Session' task to the 'FF' section of the default list correctly.
Habit streak calculation is correct for all frequency types including retroactive day toggles in SCR-06.
